Layer 2 — The System's Job (Tom's three AI capabilities)
Covers Trigger + Instantiate

Define

Auto-generate DD questions from deal type. Map to standardised templates (~90% standard, ~10% deal-specific). Create workstreams. Set up Box email address. Process initial document upload.

Powers Enrich

Retrieve

Read Box documents via daily batch. Extract findings. Match evidence to outstanding questions. Surface with confidence scores and source links. Flag conflicts with confirmed answers.

Powers Orchestrate

Orchestrate

Surface what's answered vs outstanding. Generate EOD/EOW summaries. Show what changed since last login. Flag stale items. Prioritise next actions for associates.

No AI equivalent.
Human outcome.

Why two layers: Tom's framework (Define, Retrieve, Orchestrate) tells you what to build. The deal journey (Trigger through Decide) tells you why it matters. Decide sits outside the system's scope — the product gets the work package to a state where the decision is possible, it doesn't make the decision. This is explicit in Apollo's requirements: "Won't Have — fully autonomous DD decision-making."

Clarification from v1: The Granular Workplan tab is a drill-down of the Vertical dimension — it breaks gating items into sub-questions with owners. Depth is distinct: it's the actual evidence (Box documents, call notes, findings) that answers those sub-questions. The product's core job is automating the connection: Retrieve reads depth-layer documents and matches findings back to vertical-layer questions, while Orchestrate tracks horizontal-layer progress.

Question Answer States

Each gating item / DD question moves through four possible states. Enrichment is not linear — new evidence can challenge confirmed answers.

Unanswered

Question exists but no evidence has been matched to it yet. Generated at Instantiate by Define.

→

AI-Suggested

Retrieve found evidence in Box that may answer this question. Surfaced with confidence score and source link. Awaiting human review.

→

Confirmed

Associate reviewed the AI suggestion and confirmed it as the "golden answer." This is the authoritative response. Locked by a human.

→

Challenged

New evidence found in a later batch conflicts with or updates the confirmed answer. Original answer preserved. New finding flagged for review.

Happy Path

New question gets answered

1
Deal created, 14 questions auto-generated (Unanswered)
2
Overnight batch processes 3 new Box documents
3
AI matches findings to 5 questions (AI-Suggested)
4
Associate logs in, sees morning briefing: "5 new suggestions"
5
Reviews each, confirms 4, rejects 1 (4 Confirmed, 1 back to Unanswered)
Conflict Path

New evidence challenges confirmed answer

1
"Cost Structure" confirmed last week based on management presentation
2
PwC uploads new financial DD report to Box overnight
3
AI extracts cost data that contradicts the confirmed answer
4
Question moves to Challenged — original answer preserved, new finding flagged
5
Associate reviews both, either updates the golden answer or dismisses the new finding
Additive Path

New evidence strengthens confirmed answer

1
"Competition" confirmed based on Bain market study
2
New industry report uploaded to Box
3
AI finds supporting data — consistent with confirmed answer
4
New source added to evidence chain, answer stays Confirmed
5
Associate sees "1 new supporting source" — no action required
Coverage Gap

Question remains unanswered

1
"Tech Debt" has been Unanswered for 5 days
2
No documents in Box contain relevant evidence
3
Orchestrate flags it as stale — appears in morning briefing
4
Associate sees which workstream should be providing this evidence
5
Can chase the responsible advisor directly from the dashboard
The key design challenge: The Challenged state is what makes this product different from a simple tracker. It's the "golden answer problem" from the designer brief — AI must never silently overwrite a human-confirmed answer, but it must surface when new evidence contradicts one. The UI needs to make this feel informative, not alarming. Think "new data available" not "your answer is wrong."

Batch, Not Real-Time

From Designer Brief — Data Freshness Model

AI processing runs as a daily batch. Documents uploaded during the day are processed overnight. All deal team members see the same state when they log in the next morning.

Design implication: The associate's entry point each morning is a "what changed overnight" briefing — not a live feed. The UI should feel like opening a newspaper, not watching a ticker. New suggestions, challenged answers, and stale flags all surface here.
↥

The Upload Moment

From Ilija on the internal call

When creating a new deal, there's a document upload step where AI processes the initial docs and auto-populates workstreams. This is the bridge between Instantiate and Enrich — the moment Define hands off to Retrieve.

Design implication: "New Deal" is not just a form. It's: name the deal → upload initial documents → AI generates questions and workstreams → associate reviews and adjusts. This is a multi-step creation flow, not a single click.
▶

Agent Panel Is a Utility

From Designer Brief — Module 3 scoping

Apollo has 30 existing PE agents. Module 3 provides links to them. It doesn't integrate into the work package lifecycle. Cemre explicitly flagged this as a scope creep risk — MVP is links only.

Design implication: The Agent Panel is a sidebar or drawer, not a core workflow step. It should be accessible but not central. Don't design enrichment or orchestration features around it. If it's cut from MVP, the product still works.
⚠

Scope Boundary

From Business Requirements — "Won't Have"

Conquer builds front-end only. Apollo (Jason Mather) owns back-end (Glean/Box integration). No DealCloud integration. No replacement of Box/Outlook/DealCloud as systems of record. No dependency tracking between deals.

Design implication: Every screen must assume data comes from an API that Conquer doesn't control. Design for graceful degradation — what does the UI show when the API is slow, incomplete, or returns unexpected data? The API contract is not yet defined (flagged as a gap in Tom's alignment doc).
Why this tab exists: The work package framework describes the "what" and "why." These constraints describe the "but also" — the real-world boundaries that will shape every screen Ilija designs. Each one traces back to a specific source document or conversation.
